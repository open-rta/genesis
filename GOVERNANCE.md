# Open RTA Governance

## Purpose
This document defines how Open RTA Genesis evolves while preserving neutrality, transparency, and implementation independence.

## Change proposal process
Changes are proposed through documented change requests that include:
- problem statement
- impacted files and sections
- rationale
- compatibility impact
- evidence or examples

Detailed process guidance is in [governance/proposal-process.md](governance/proposal-process.md).

## Change classes
- **Clarifying**: improves wording without changing normative meaning.
- **Normative**: changes requirements, constraints, or conformance expectations.
- **Breaking**: introduces incompatibility with prior compliant artifacts or interpretations.

## Law evolution
Law-level changes require explicit rationale, compatibility analysis, and broader review than ordinary clarifications. See [governance/law-change-process.md](governance/law-change-process.md).

## Review expectations
Until a formal governance body is established, review should include maintainers and cross-runtime stakeholders with no single runtime privileged to define shared law.

## Transparency expectations
All decisions should be recorded in versioned documents, with public rationale and explicit change classification.

## Neutrality expectation
No runtime-specific interest should redefine Open RTA law for all participants. Runtime-specific practices belong in implementation repositories, not in Genesis law text.
